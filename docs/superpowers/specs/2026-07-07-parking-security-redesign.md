# 주차현황 조회 — 보안 재설계 (Spec)

- 작성일: 2026-07-07
- 상태: 설계 확정 (구현 대기)
- 범위: 기존 "주차현황 조회" 기능의 **데이터/비밀번호 노출 차단**을 위한 아키텍처 재설계
- 관련 문서: `2026-07-07-parking-status-design.md` (원 기능 설계). 이 문서는 그 위에 **접근/보안 계층만** 교체한다.

---

## 1. 목적 / 배경

기존 구현은 **순수 정적 사이트**로, 브라우저가 구글시트 CSV 두 개(차량/설정)를 직접 `fetch`해서 클라이언트에서 비밀번호 비교·차량 매칭까지 전부 수행한다.

그 결과 **레포가 public이든 private이든 무관하게** 다음이 노출된다:
- 배포된 사이트의 JS를 열면 두 CSV 주소가 그대로 보인다.
- 그 주소로 직접 들어가면 비밀번호 화면을 거치지 않고도 **전체 입주민 동/호/차량번호**와 **4자리 비밀번호 값 자체**를 평문으로 볼 수 있다.

즉 현재 비밀번호 게이트는 "우연히 링크 눌러 들어오는 것"만 막을 뿐 진짜 데이터 보호가 아니다.

**이번 목표**: 동/호, 차량번호, 비밀번호가 외부에 노출되지 않게 한다. 개발자도구/직접 URL 접근으로도 원본 데이터나 비밀번호 값 자체를 볼 수 없어야 한다.

## 2. 위협 모델 (확정)

- **막는 대상 = 공용 비밀번호를 모르는 외부인/일반 대중.** (우연히 링크 눌러온 사람, 검색엔진 크롤러, 공개 깃허브 레포를 열어본 사람.) 이들에게는 데이터·비밀번호 값·시트 주소가 **전부** 안 보여야 한다. 이것이 이번 요구사항이다.
- **막지 않는(막을 수 없는) 대상 = 공용 비밀번호를 아는 사람.** 검색 기능의 본질이 "차량번호 → 동/호 알려주기"이므로, 비밀번호를 아는 사람은 뒷 4자리(0000~9999)를 넣어보며 등록 차량의 동/호를 열거할 수 있다. 이 한계는 luigi와 합의됨(입주민 간 방지는 이 기능과 양립 불가).
- 근본 제약: **깃허브 페이지 등 순수 정적 호스팅은 원리상 비밀을 숨길 수 없다.** 클라이언트로 보내는 모든 것은 개발자도구로 노출된다. 따라서 서버측 구성요소가 반드시 하나 필요하다.

## 3. 아키텍처

프론트(정적)와 서버함수(프록시)를 **같은 공개 깃허브 레포**에 두고 **Cloudflare Pages**로 배포한다. Cloudflare Pages가 레포에 연결되어 git push마다 자동배포하며, `functions/` 디렉터리의 서버함수도 같은 도메인에서 서빙한다(→ **프론트와 API가 동일 출처, CORS 불필요**).

```
index.html, styles.css, app.js   # 프론트: 구글시트 직접 접근/매칭 로직 전부 제거
config.js                        # API 경로 상수만 (비밀 없음)
functions/api/search.js          # 서버함수: 비번검증 + 매칭. 시트 주소는 env 비밀
match.js                         # 순수 매칭 함수 (서버함수가 import해서 재사용)
match.test.js                    # match.js 자체 점검 (node 실행)
```

**데이터 흐름**
1. 브라우저는 구글시트를 **직접 보지 않는다.** 서버함수 `POST /api/search` 하나만 호출한다.
2. **서버함수만** 두 CSV 주소(차량/설정)를 안다 — Cloudflare Pages **암호화 환경변수** `VEHICLES_CSV_URL`, `SETTINGS_CSV_URL`.
3. 서버함수가 매 요청 때 시트를 서버에서 읽어 비밀번호 검증 + 차량 매칭을 하고 **결과만** 돌려준다.
4. 클라이언트로는 원본 데이터·비밀번호 값·시트 주소가 **한 번도** 나가지 않는다.

**컴포넌트 경계**
- `config.js` — API 경로(`/api/search`) 상수만. 비밀 없음.
- `match.js` — 순수 함수 `normalizePlate`, `lastFourDigits`, `findMatches`. fetch/DOM 없음 → 단독 테스트 가능. **서버함수가 재사용**(기존 로직 그대로).
- `functions/api/search.js` — 부작용 담당: 시트 fetch·파싱·캐싱, 비번 검증, 매칭 호출, rate-limit, 최소 응답 생성.
- `app.js` — 게이트 UI, `/api/search` 호출, 결과 렌더. 시트 파싱·클라이언트 비번비교·현황 렌더 **제거**.

## 4. 서버함수 요청/응답

**요청**: `POST /api/search`, 본문 `{ "password": "1234", "query": "3456" }`
- `query`가 비어있거나 없으면 = "비밀번호만 확인"(게이트 진입용).

**응답** (원본 노출 최소화 — 전체 목록은 어떤 경우에도 통째로 내려주지 않는다):

| 상황 | HTTP | 본문 |
|------|------|------|
| 비번 확인만 성공 | 200 | `{ "status": "ok" }` |
| 검색 0건 | 200 | `{ "status": "none" }` |
| 검색 1건 | 200 | `{ "status": "one", "dong": "101", "ho": "1203", "type": "" }` |
| 검색 여러 건 | 200 | `{ "status": "many", "results": [ { "dong": "101", "ho": "1203", "type": "" }, ... ] }` |
| 비번 틀림 | 401 | `{ "error": "bad_password" }` |
| 시도 과다 | 429 | `{ "error": "rate_limited" }` |
| 시트 로드 실패 | 502 | `{ "error": "upstream" }` |

- **여러 건일 때도 차량번호는 보내지 않는다.** 동/호(+차종 `type`)만. 동/호만으로 서로 다른 세대가 구분되므로 차량번호 없이도 식별에 충분하다. (기존 설계는 다건 시 괄호로 차량번호를 표시했으나, 이번엔 노출 최소화를 위해 제거.)
- `type`은 차종 문자열(예: `이륜` 또는 빈 문자열). 민감정보가 아니며 🚗/🛵 표시에 쓰인다. 이모지 매핑은 클라이언트가 담당(`typeEmoji`).

## 5. 비밀번호 흐름 (상태 없음)

- 게이트에서 비번 입력 → `/api/search`에 비번만 보내 확인(`query` 빈 값) → 200이면 그 비번을 **sessionStorage**에 저장(그 세션 동안 재입력 없음) → 이후 검색마다 비번+검색어를 함께 전송.
- 서버는 매 요청마다 설정시트의 비번과 대조. **별도 토큰/세션 서버 없음**(45세대 규모엔 과함).
- 세션 중 관리자가 비번을 바꾸면, 다음 요청이 401 → 클라이언트가 sessionStorage 플래그를 비우고 **게이트로 되돌림**.
- 비번은 코드에 하드코딩하지 않는다(설정시트가 유일 출처). 백업 비번도 두지 않는다.

## 6. 무차별대입 방지

이제 비밀번호가 **유일한 보안 경계**이므로(기존엔 데이터가 어차피 공개라 사실상 장식), 방어가 없으면 4자리(1만 조합)를 서버에 1만 번 요청해 자동 돌파할 수 있다.

- 서버함수가 **IP당 실패 시도 횟수를 제한**한다(예: 분당 10회 초과 시 429). 이러면 자동 전수시도가 시간 단위로 늦춰져 공용비번+소규모 상황엔 충분하다.
- 구현 메커니즘: **Cloudflare 기본 rate-limiting 바인딩 우선**, 배포 시점에 Pages Functions에서 못 쓰면 **KV 네임스페이스 카운터(IP 키, 짧은 TTL)로 대체**. 임계값(분당 N회)은 구현 단계에서 확정.
- CAPTCHA 등은 넣지 않는다(과설계).
- **정직한 한계**: 4자리 공용비번 자체가 약한 보안이라 "완벽 차단"이 아니라 "자동 전수시도를 비현실적으로 느리게" 수준이다. luigi가 이 수준으로 충분하다고 합의함(4자리 유지, 자릿수 확대 안 함).

## 7. 캐싱 / 데이터 갱신

- 서버함수가 시트 CSV를 매 요청 fetch하되 **짧게 캐싱(약 60초)**. 구현은 Workers isolate의 모듈 전역 변수 + 타임스탬프면 충분(콜드 isolate에서 가끔 재fetch되는 건 무해). 별도 캐시 클래스/스토어 불필요.
- 구글시트에서 데이터·비번을 수정하면 최대 ~1분 뒤 반영(기존 정적 버전의 CSV 발행 지연과 비슷). **관리자가 시트만 고치면 되는 편의성은 그대로 유지.**

## 8. 현황 탭

- **계속 끔.** 검색 기능만 제공한다. 전체 목록을 클라이언트로 내리는 경로 자체를 없애 차량번호 노출 위험을 원천 제거한다. (탭이 검색 하나만 남으므로 구현자는 탭 바 UI를 생략해도 된다.)
- 나중에 다시 켜고 싶으면, 서버가 **동/호만(차량번호 제외)** 내려주는 버전으로 별도 논의한다. (인증된 클라이언트라도 전체 차량번호를 통째로 내려주면 "차량번호 노출 안 되게"와 충돌하므로.)

## 9. 에러 처리

- 시트 로드 실패(네트워크/미발행): 서버 502 → 게이트 화면은 `확인 정보를 불러오지 못했습니다. 다시 시도해주세요.`, 검색은 `차량 정보를 불러오지 못했습니다. 잠시 후 다시 시도해주세요.` + [다시 시도].
- 비번 틀림: 게이트에 `비밀번호가 올바르지 않습니다.`
- 시도 과다(429): 게이트/검색에 `잠시 후 다시 시도해주세요.`
- 설정시트에 4자리 숫자 비번이 없으면: 서버가 502(`upstream`) 취급.
- 표시 문구는 기존 설계 문서의 문자열을 그대로 사용.

## 10. 배포 시 반드시 할 것 (보안상 중요)

현재 `config.js`에 들어있는 **기존 시트 CSV 두 주소는 이미 "웹에 게시"된 공개 bearer URL**이다. 링크만 알면 누구나, 영구적으로(게시 중지 전까지) 읽을 수 있다. 그대로 두면 서버를 우회하는 전체 노출 경로가 남는다.

1. 구글시트에서 기존 게시를 **게시 중지 후 다시 게시**해 **새 주소**를 발급받고, **기존 주소는 폐기**한다. (기존 주소는 이미 코드/브라우저 히스토리 등에 남아 있을 수 있으므로 재사용 금지.)
2. 새 주소 2개는 **Cloudflare Pages 환경변수(암호화)에만** 넣는다. **깃허브 레포/코드에는 절대 커밋하지 않는다.**
3. 공개 레포에 시트 주소·비밀번호·데이터가 **하나도** 들어가지 않게 한다. (`config.js`에는 API 경로만.)

**더 강한 대안(지금은 안 함)**: 시트를 아예 비공개로 두고 구글 API+서비스계정으로 서버가 읽기 — public bearer URL 자체를 없앨 수 있으나 설정이 복잡해 "최대한 단순" 원칙상 이번엔 채택하지 않는다. 위 방식으로 요구는 충족되며, 필요해지면 나중에 전환한다.

## 11. 배포 절차 (요약)

1. Cloudflare 무료 계정 생성.
2. Cloudflare Pages 프로젝트를 **공개 깃허브 레포**에 연결(빌드 명령 없음, 정적, 출력 루트). `functions/`가 자동으로 라우팅됨.
3. Pages 프로젝트 환경변수(암호화)에 `VEHICLES_CSV_URL`, `SETTINGS_CSV_URL`(새 발급 주소) 설정.
4. rate-limit용 리소스(rate-limiting 바인딩 또는 KV 네임스페이스) 바인딩.
5. git push → 자동 배포 → 아래 검증.

## 12. 테스트 / 검증

- **자동 자체 점검** `match.test.js`(`node match.test.js`, assert 기반, 프레임워크 없음) — 기존 5케이스 유지:
  1. 전체 번호 완전일치 → 1건
  2. 뒷 4자리 입력 → 해당 차량 매칭
  3. 공백 포함 입력도 매칭(정규화)
  4. 없는 번호 → 0건
  5. 서로 다른 세대가 같은 뒷 4자리 → 여러 건
- **배포본 육안 검증**(구현자):
  1. 게이트 통과 → 검색 정상/미등록 렌더.
  2. 개발자도구 **Network 탭**에서 어떤 응답에도 원본 차량번호·비밀번호 값·시트 주소가 없음을 확인.
  3. 페이지 소스/JS에 시트 주소·비밀번호·데이터가 없음을 확인.
  4. 잘못된 비번을 반복 요청 → 429 확인.
  5. 모바일 뷰포트 확인.
- (well-formed ≠ correct: 실제 배포본에서 위를 직접 눈으로 본 뒤 완료 처리.)

## 13. 범위 밖 / 향후

- 진짜 개인화 인증(세대별 로그인), 현황 탭 재도입, 실시간 반영, 관리자 폼: 이번 범위 밖.
- 구글 API+서비스계정 기반 비공개 시트 전환: 필요해질 때 §10의 강한 대안으로.
- 공지사항·커뮤니티 등 다른 기능: 이번 범위 밖.

## 14. 구현자 유의 (요약)

- **선행 확인**: Cloudflare Pages Functions에서 서버측 `fetch`로 구글시트 CSV를 읽을 수 있는지 확인(서버→구글 요청이라 CORS 무관, 정상 동작 예상).
- `match.js`는 순수 함수로 유지하고 서버함수에서 재사용(중복 구현 금지). 단 서버함수(Pages Function)는 ES 모듈이고 현재 `match.js`는 node 테스트용 `module.exports`(CommonJS)만 있으므로, `export`도 함께 노출하거나 테스트 실행 방식을 맞춰 양쪽에서 쓰이게 조정할 것.
- 시트 주소·비번·데이터는 **코드/레포에 절대 넣지 않는다.** 오직 Cloudflare 환경변수.
- 응답에서 차량번호를 절대 내보내지 않는다(동/호/차종만).
- 표시 문구는 기존 설계 문서 문자열을 그대로, 한 곳에 모아두면 좋다.
