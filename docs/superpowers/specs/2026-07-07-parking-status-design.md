# 주차현황 조회 — 설계 문서 (Spec)

- 작성일: 2026-07-07
- 상태: 설계 확정 (구현 대기)
- 범위: 입주민 전용 모바일웹의 **첫 기능 "주차현황 조회"** 하나에 한정

---

## 1. 목적 / 배경

입주민(오피스텔/빌라)이 주차장에서 본 차량이 **입주민 차량인지**를 모바일에서 빠르게 확인하기 위한 웹.
같은 사이트에 이후 공지사항·커뮤니티 등이 붙을 수 있으나 **이번 범위 밖**(YAGNI).

## 2. 확정된 요구사항 (인터뷰 결과)

- 화면은 탭 2개: **검색 / 현황**.
- **검색**: 차량번호를 입력 → 등록 차량이면 "○○○동 ○○○호 이웃입니다." / 미등록이면 안내 문구.
- **현황**: 등록 차량 전체 목록을 표로 표시(동·호·차량번호 전체 노출).
- **데이터**: 동 / 호 / 차량번호(전체 저장). **세대당 1대 원칙**(1세대 1주차).
- **DB 관리**: 관리자(luigi)가 **구글시트**에서 직접 수정 → 사이트에 반영.
- **접근 제어**: 로그인 없음. **4자리 숫자 공용 비밀번호**로 게이트. 비밀번호 값은 **구글시트에서 관리**(코드 하드코딩 금지).
- **규모**: 20~50세대.
- **비용/스택**: 완전 무료 + 최대한 단순 + 정적 호스팅.

## 3. 아키텍처 & 파일 구조

빌드 도구 없는 **순수 정적 사이트**(바닐라 HTML/CSS/JS). 프레임워크·서버·신규 라이브러리 없음.

```
index.html      # 마크업 + 탭 2개
styles.css      # 스타일 (모바일 우선)
config.js       # 상수 2개: 차량 CSV 주소, 설정 CSV 주소
match.js        # 순수 로직: 번호 정규화 + 검색 매칭 (DOM 의존 없음, 테스트 대상)
app.js          # CSV fetch/파싱, 비밀번호 게이트, 렌더링, 이벤트
match.test.js   # match.js 자체 점검 (node로 실행)
```

**컴포넌트 경계**
- `config.js` — 값만: `VEHICLES_CSV_URL`, `SETTINGS_CSV_URL`.
- `match.js` — 순수 함수 `normalizePlate(s)`, `findMatches(vehicles, query)`. fetch/DOM 없음 → 단독 테스트 가능.
- `app.js` — 부작용 담당: CSV 두 개 fetch·파싱, 비밀번호 확인, 검색 탭·현황 탭 렌더, 탭 전환.

**데이터 흐름**: 접속 → 설정 CSV로 비밀번호 확인(게이트) → 차량 CSV 로드해 메모리 배열 보관 → 검색/현황 두 탭이 같은 배열을 사용.

## 4. 데이터 모델 (구글시트)

같은 스프레드시트에 **탭 2개**.

**(1) '차량' 탭** — 열 3개, 1행은 제목(무시)

| 동 (A) | 호 (B) | 차량번호 (C) |
|--------|--------|--------------|
| 101 | 1203 | 12가3456 |
| 102 | 803 | 34나7890 |

- 차량번호는 **전체**로 저장.
- 세대당 1대 원칙 → 보통 한 세대 = 한 줄. (예외로 2대가 생겨도 줄만 추가하면 그대로 동작 — 구조 변경 불필요.)
- 빈 줄 / 차량번호 없는 줄은 무시.

**(2) '설정' 탭** — 4자리 비밀번호 저장

- **A1 셀에 4자리 숫자 비밀번호만** 입력(예: `1234`).
- **주의(중요)**: 이 셀은 반드시 **일반텍스트 서식**(서식 > 숫자 > 일반텍스트)으로 둘 것. 숫자 서식이면 `0119` 같은 **앞자리 0이 사라져** CSV에 `119`로 나오고 로그인이 전부 실패한다. (텍스트 서식이 어려우면 앞자리 0 없는 비밀번호를 쓰도록 안내.)
- 앱은 설정 CSV에서 **첫 번째 4자리 숫자**를 비밀번호로 사용.

**발행 방법**: 두 탭 각각 "파일 > 공유 > 웹에 게시 > (해당 시트 선택) > CSV"로 발행 → 나온 두 주소를 `config.js`의 `VEHICLES_CSV_URL` / `SETTINGS_CSV_URL`에 넣음. 이후엔 시트만 고치면 됨(코드 수정 불필요).

**CSV 파싱**: 동/호/차량번호·비밀번호에는 쉼표·따옴표가 없으므로 줄바꿈+쉼표 단순 분리로 충분.
`// ponytail: 단순 split 파싱. 자유 텍스트(비고 등) 열이 추가되면 정식 CSV 파서로 교체.`

## 5. 검색 탭 UX & 매칭 규칙

- 구성: 입력창 1개 + [검색] 버튼(엔터로도 검색).
- **정규화**: 입력·저장 번호 모두 **공백 제거** 후 비교.
- **매칭 규칙** (`findMatches`):
  1. 공백 제거한 입력이 차량번호(공백 제거)와 **완전히 같으면** 매칭.
  2. 입력이 **정확히 4자리 숫자**이면, 차량번호에서 **숫자만 뽑았을 때 끝 4자리**가 입력과 같은 경우도 매칭(차 보고 뒷자리만 기억하는 상황 대응). 예: `12가3456` → 끝 4자리 `3456`.
  - 세대당 1대이므로 전체번호 검색은 최대 1건. **뒷자리 검색은 서로 다른 세대에서 여러 건**이 나올 수 있음.
- **결과 표시**:
  - **1건**: `{동}동 {호}호 이웃입니다.`
  - **여러 건**: `여러 대가 검색되었습니다:` + 각 줄 `{동}동 {호}호 ({차량번호})`
  - **0건**: `등록되지 않은 차량입니다. 방문차량인지 미등록 차량인지 채팅방에서 확인해주세요.`
    - 아래에 **채팅방 링크 버튼 자리**를 마크업만 두고 지금은 숨김. 나중에 링크만 넣으면 노출(섹션 9 참조).
- 검색 결과에는 원칙적으로 번호를 다시 표시하지 않음(여러 건일 때만 구분 위해 괄호로 표시).

## 6. 현황 탭 UX

- 등록 차량 **전체 표**: 열 = **동 / 호 / 차량번호(전체)**.
- 정렬: 동 → 호 **숫자 오름차순**. 동/호가 숫자가 아니면(예: `가동`, `A동`) 문자열 정렬로 폴백.
- 상단에 `총 N대` 표시.
- 모바일에서 표가 넘치면 **가로 스크롤** 컨테이너로. 별도 필터창 없음(검색 탭이 그 역할).
- 등록 0대: `등록된 차량이 없습니다.`

## 7. 접근(비밀번호) 처리

- 첫 진입 시 **비밀번호 입력 화면** → 설정 CSV에서 읽은 4자리와 비교.
  - 일치: 본문(탭 2개) 표시. **그 세션 동안 재입력 없음**(sessionStorage 플래그).
  - 불일치: `비밀번호가 올바르지 않습니다.`
- 값은 코드가 아니라 **구글시트 '설정' 탭**에 있음 → 관리자가 시트에서 바꾸면 반영.
- **코드에 백업 비밀번호를 두지 않음**(우회 방지). 설정 CSV를 못 읽으면 로그인 화면에 오류+다시 시도(섹션 8).
- **정직한 보안 한계**: 설정/차량 CSV 주소와 프론트 코드는 공개형이라, 작정하고 찾으면 데이터·비밀번호를 볼 수 있음. 즉 이 게이트는 **"우연히 아무나 들어오는 것 막기" 수준**이며 진짜 인증이 아님 — 애초에 현황 탭이 입주민에게 전체 공개인 성격이라 현재 요구엔 부합. 진짜 로그인/접근제어가 필요해지면 백엔드(예: Supabase) 재검토.

## 8. 에러 처리 & 캐시

- **차량 CSV 로드 실패**(네트워크/미발행): `차량 정보를 불러오지 못했습니다. 잠시 후 다시 시도해주세요.` + [다시 시도].
- **설정 CSV 로드 실패**: 로그인 화면에 `확인 정보를 불러오지 못했습니다. 다시 시도해주세요.` + [다시 시도].
- 차량 0대: 현황 탭 `등록된 차량이 없습니다.`, 검색은 항상 미등록으로.
- 차량번호가 빈 줄은 건너뜀.
- **캐시 주의**: 구글 CSV 발행본은 시트 수정이 화면에 반영되기까지 **최대 수 분 지연**될 수 있음(실시간 아님). 페이지는 열 때마다 새로 fetch.

## 9. 기술 스택 & 호스팅

- 바닐라 HTML/CSS/JS(빌드 없음). CSV는 브라우저 `fetch`로 직접 로드.
- 호스팅: **GitHub Pages 또는 Cloudflare Pages**(둘 다 무료 정적 호스팅). 이후 공지·커뮤니티는 탭/페이지 추가로 확장.
- 저장소는 아직 git 초기화 전 — **git init 여부·호스팅 선택은 구현 단계에서 결정**.

## 10. 테스트 / 검증 방식

- **자동 자체 점검** `match.test.js` (`node match.test.js`, assert 기반, 프레임워크 없음) — 최소 5가지:
  1. 전체 번호 완전일치 → 1건
  2. 뒷 4자리 입력 → 해당 차량 매칭
  3. 공백 포함 입력도 매칭(정규화)
  4. 없는 번호 → 0건
  5. 서로 다른 세대가 같은 뒷 4자리 → 여러 건
- **브라우저 육안 확인**(구현자): 로그인 게이트 → 검색 정상/미등록 → 현황 표 렌더 → 모바일 뷰포트. (well-formed ≠ correct: 실제 렌더 확인 후 완료.)

## 11. 범위 밖 / 향후

- 공지사항, 커뮤니티 등 다른 기능: 이번 스펙 밖. 사이트가 정적으로 커질 수 있게만 열어둠.
- 채팅방(카카오 오픈채팅 등) **링크**: 지금은 문구만, 버튼 자리만 마련. 링크 확정되면 `config.js`에 추가해 노출.
- 실시간 반영/관리자 폼/진짜 인증: 필요해질 때 백엔드 도입 재검토.

## 12. 구현자 유의 (요약)

- **구현 1단계(선행 확인)**: 실제 발행된 CSV 주소를 **브라우저 `fetch`로 CORS 동작 확인**. 이 접근법 전체가 여기에 달려 있음. `.../pub?output=csv`가 막히면 **`gviz/tq?tqx=out:csv` 엔드포인트를 폴백**으로 사용(같은 시트, 클라이언트 fetch 가능). 안 되면 접근법 A 재검토 후 luigi에 보고.
- `match.js`는 순수 함수로 유지(테스트 가능).
- `config.js`에는 **주소 2개**만. 비밀번호·데이터 값은 코드에 넣지 않음.
- 표시 문구는 위 스펙의 문자열을 그대로 사용(luigi가 추후 문구 수정 가능하도록 한 곳에 모아두면 좋음).
